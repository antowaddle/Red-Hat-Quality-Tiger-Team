# Konflux Build Simulator — odh-dashboard

Generated by the **Konflux Build Simulator** skill from analysis of [opendatahub-io/odh-dashboard](https://github.com/opendatahub-io/odh-dashboard).

## Why This Matters

**111 bugs** have been logged in [RHOAIENG](https://redhat.atlassian.net/issues?filter=-4&jql=project%20%3D%20RHOAIENG%20AND%20type%20%3D%20Bug%20and%20description%20~%20crash) where crashes caused the DSC (DataScienceCluster) to go **NotReady**, which causes Konflux pipelines to fail and blocks entire teams. Build failures that slip past PR validation are a major contributor to this pattern — they only surface in Konflux or production, by which point the damage is done.

The Konflux Build Simulator catches these failures **at PR time** — before merge, before Konflux, before anyone is blocked.

## What It Validates

1. **Docker build** with production `BUILD_MODE=RHOAI` (same args as `.tekton/odh-dashboard-push.yaml`)
2. **Module Federation host artifact** — verifies `remoteEntry.js` exists at the correct path (`/usr/src/app/frontend/public/`)
3. **Container runtime** — starts the image and confirms it responds on port 8080
4. **Kustomize manifests** — builds the three canonical roots: `manifests/rhoai/addon`, `manifests/rhoai/onprem`, `manifests/odh`

## Bugs Found During Analysis

- **Module Federation path mismatch** — existing simulator checks `/opt/app-root/src/packages/...` but the Dockerfile uses `/usr/src/app`. This generated workflow uses the correct path.
- **Health endpoint unusable standalone** — `/api/health` requires a Kube context. This workflow uses `GET /` instead.
- **Manifest validation scope inconsistency** — two different existing workflows validate different kustomize roots. This workflow aligns with the three official roots.

## Usage

### In CI
Copy `workflows/konflux-build-simulator.yml` to `.github/workflows/` in the odh-dashboard repository.

### Locally
```bash
chmod +x scripts/validate-build.sh
./scripts/validate-build.sh
```
