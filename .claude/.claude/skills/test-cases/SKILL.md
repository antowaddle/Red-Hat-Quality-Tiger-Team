# Test Case Generator

Generate individual test case specification files from an existing test plan.

## Inputs

### Auto-detection
If a TestPlan.md was just generated by the `/test-plan` skill in this session, use its feature directory automatically — no user input needed. Proceed directly to Step 1.

### Manual input
If no TestPlan.md was recently generated, ask the user for:
1. **Feature directory**: Path to the feature directory containing `TestPlan.md` (e.g., `mlmd_removal`)

## Process

### Step 1: Read the Test Plan

1. Read `<feature_dir>/TestPlan.md` using the Read tool
2. Extract:
   - Section 4 (Endpoints/Methods Under Test) — the full list of what needs test coverage
   - Section 2 (Test Strategy) — test levels, types, priorities to guide test case depth
   - Section 3 (Test Environment) — preconditions and test data requirements
   - Section 5.2 (Test Case Naming Convention) — the `TC-<CATEGORY>-<NUMBER>` prefixes and their meanings
   - Section 1.2 (Scope) — in-scope vs out-of-scope boundaries

### Step 2: Read the Test Case Template

1. Read the template from `.claude/skills/test-cases/templates/test-case-template.txt` using the Read tool
2. Follow this template structure for every generated test case
3. Omit optional sections (Preconditions, Test Data, Expected Response, Validation) when they are empty or not applicable
4. Always leave **Automation Status** and **Notes** as placeholders — they are filled later in the process

### Step 3: Design Test Cases

For each category prefix from Section 5.2, design test cases that:
- Cover every endpoint/method from Section 4
- Include positive, negative, and boundary scenarios (per Section 2.2)
- Assign priorities (P0/P1/P2) following the criteria in Section 2.3
- Stay strictly within the scope defined in Section 1.2 — do NOT create test cases for out-of-scope items

### Step 4: Generate Files

1. Create `<feature_dir>/test_cases/` directory if it doesn't exist: `mkdir -p <feature_dir>/test_cases`
2. Generate one `TC-<CATEGORY>-<NUMBER>.md` file per test case following the template
3. Generate `<feature_dir>/test_cases/INDEX.md` with:
   - Quick stats (total test cases, P0/P1/P2 counts)
   - Test cases organized by category in tables with columns: Test Case ID (linked), Title, Priority
   - Links to the parent TestPlan.md

### Step 5: Update the Test Plan

Update `<feature_dir>/TestPlan.md`:
1. **Section 5** — Update the note to reflect test cases have been generated
2. **Section 5.1** — Fill in the Test Case Organization table with category, test case ranges, and priority distribution
3. **Section 8.1** — Fill in the Test Case Summary table with counts per category
4. **Section 8.2** — Fill in the Test Cases column with TC IDs mapped to each endpoint and set Coverage status

## Guidelines

- Do NOT invent requirements not present in the test plan
- If the test plan is ambiguous about what to test, ask the user via AskUserQuestion
- Each test case file must be self-contained — a tester should be able to execute it without reading the test plan
- Use realistic test data, not placeholder values like "example.com" or "test123"
