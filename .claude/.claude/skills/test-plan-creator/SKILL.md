# Test Plan Generator

Generate a complete test plan for a RHOAI feature based on a refined strategy, its source RFE, and optionally an ADR document for additional technical depth.

## Inputs

### Auto-detection
If strategy and RFE files were just generated by `/strat.create` or `/strat.refine` in this session, use them automatically — no user input needed. Proceed directly to Step 1.

### Manual input
If no strategy/RFE files are available from the current session, ask the user for:

#### Required
1. **Strategy Jira key**: A RHAISTRAT Jira key (e.g., `RHAISTRAT-400`)
2. **Source RFE Jira key**: A RHAIRFE Jira key (e.g., `RHAIRFE-1595`)

#### Optional
3. **ADR file path**: Local path to an exported ADR document (markdown, text, or PDF) for additional technical detail (API specs, data models, etc.)
4. **ADR link**: Google Doc URL (stored as reference in metadata, not fetched)
5. **Feature directory name**: snake_case name for the feature directory (e.g., `mcp_catalog`). If not provided, derive from the feature name.

## Process

### Step 1: Gather Information

1. **Strategy**: If a Jira key was provided, fetch it using `jira_get_issue` with `fields='*all'`. If auto-detected, read the local file from `artifacts/strat-tasks/`.
2. **Source RFE**: If a Jira key was provided, fetch it using `jira_get_issue` with `fields='*all'`. If auto-detected, read the local file from `artifacts/rfe-tasks/`.
3. **ADR** (if provided): Read the ADR file for additional technical detail (API endpoints, data models, implementation specifics).

### Step 2: Analyze

From the strategy, RFE, and optionally ADR, identify:
- Feature scope (in-scope vs out-of-scope)
- API endpoints or methods under test
- Test categories needed (e.g., API, Data Validation, Error Handling, Security, Regression, Loading/Config)
- Test environment requirements
- Risks and mitigations

The strategy's technical approach is the primary source for identifying what to test. The RFE provides the business context and acceptance criteria. The ADR, when available, provides the deepest technical detail.

Do NOT invent or assume requirements not present in the source documents. In particular:
- Do NOT fabricate API endpoints, paths, methods, or query parameters that are not explicitly mentioned in the strategy, RFE, or ADR
- If the source documents do not specify concrete endpoints, leave Section 4 with a placeholder noting that endpoint details are pending and ask the user via AskUserQuestion whether they can provide an API specification or ADR
- If the scope is ambiguous, ask the user for clarification using AskUserQuestion

### Step 3: Generate Files

1. Create the feature directory using Bash: `mkdir -p <feature_name>/test_cases`
2. Read the template from `${CLAUDE_SKILL_DIR}/test-plan-template.md` using the Read tool
3. Generate `<feature_name>/TestPlan.md` by filling in the template with the gathered information. Follow the template structure exactly — do not add, remove, or reorder sections.
4. For Section 8.2 ({Endpoint/Method} Coverage): fill in the Endpoint column using the endpoints identified in Section 4. Leave the Test Cases and Coverage columns empty — they will be filled later in the process.
5. Generate `<feature_name>/README.md` with:
   - Feature name and one-line description
   - Links to Jira strategy/RFE, ADR (if provided)
   - Link to TestPlan.md
   - Brief mention of where automated tests will be implemented

### What this skill does NOT do

- Does NOT generate individual test case files
- Does NOT fetch child stories under the epic
- Does NOT fetch the Google Doc ADR — it reads a local file only
- Section 5 (Test Cases): left as placeholder — to be filled later in the process
- Section 8.1 (Test Case Summary): left as placeholder — to be filled later in the process
- Section 8.2 Test Cases and Coverage columns: left empty — to be filled later in the process
